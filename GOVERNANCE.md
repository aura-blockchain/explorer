# AURA Explorer Governance

## Overview

The AURA Explorer is maintained by the AURA core team with community contributions welcome. This document outlines how decisions are made.

## Roles

### Maintainers
- Merge pull requests
- Review code and proposals
- Set project direction
- Manage releases

### Contributors
- Submit pull requests
- Report issues
- Participate in discussions
- Review code

## Decision Making

### Minor Changes
Bug fixes, documentation updates, and small improvements can be merged with one maintainer approval.

### Significant Changes
Feature additions or architectural changes require:
1. GitHub issue or discussion for proposal
2. Community feedback period (7 days minimum)
3. Two maintainer approvals
4. Passing CI checks

### Breaking Changes
Changes that break backward compatibility require:
1. RFC (Request for Comments) issue
2. Extended discussion period (14 days)
3. Migration path documented
4. Unanimous maintainer approval

## Proposing Changes

1. Open a GitHub issue describing the change
2. Label appropriately (enhancement, bug, etc.)
3. Engage with feedback
4. Submit PR when there's consensus

## Release Process

- Semantic versioning (MAJOR.MINOR.PATCH)
- Changelog updated with each release
- Releases tagged in git
- Docker images published

## Disputes

If consensus cannot be reached:
1. Extended discussion period
2. Seek input from AURA core team
3. Final decision by project lead

## Amendments

This governance document may be amended through the same process as significant changes.

## Contact

- GitHub Issues: Primary communication channel
- Discord: Real-time discussion
- Email: For private matters
